## Root Cause: All failures trace to ONE issue — CSRF

**Evidence:** Look at which tests pass vs fail:

| Test | Method | Result |
|------|--------|--------|

The pattern is exact: **every GET passes, every state-changing method fails**. CSRF middleware only checks POST/PATCH/PUT/DELETE.

The "wrong password" test passes because `assertGuest()` is true regardless of whether login failed due to bad password OR due to a 419 — user stays unauthenticated either way.

---

## Why this happens in Laravel 11+

In **Laravel 10 and below**, the app shipped with `app/Http/Middleware/VerifyCsrfToken.php` — a class you could override. In **Laravel 11+**, that class is gone. CSRF is now registered internally via `ValidateCsrfToken` in the `web` middleware group, configured only through `bootstrap/app.php`.

`TestCase.php` is completely bare — no CSRF disabling. `bootstrap/app.php`'s `withMiddleware` callback is empty. So CSRF runs fully in tests.

With `SESSION_DRIVER=array`, each test request gets a fresh empty session — no CSRF token stored. The request body has no `_token`. `hash_equals(null, null)` = `false` → **419 on every write request**.

The login/register failures say "user not authenticated" not "419" because the test asserts `assertAuthenticated()` before `assertRedirect()` — the 419 is the underlying cause but the first assertion to fail is the auth check.

---

## Proposed Fix

One change in `bootstrap/app.php` — disable CSRF when running unit tests:

```php
->withMiddleware(function (Middleware $middleware): void {
    if (app()->runningUnitTests()) {
        $middleware->web(remove: [
            \Illuminate\Foundation\Http\Middleware\ValidateCsrfToken::class,
        ]);
    }
})
```

`app()->runningUnitTests()` is Laravel's official flag — returns `true` only during PHPUnit runs, so production CSRF protection is untouched.

## Summary of what was found and fixed

**Root cause:** Laravel 13 renamed the CSRF middleware from `VerifyCsrfToken` → `PreventRequestForgery`. The new class has a built-in `runningUnitTests()` bypass, but it requires `app()->runningInConsole() && $this['env'] === 'testing'`. The `bootstrap/app.php` approach to disable it evaluated too early in the boot cycle for the condition to reliably fire.

**Fix applied:** `tests/TestCase.php` — `withoutMiddleware(PreventRequestForgery::class)` in `setUp()`. This is explicit, runs after the app is created, and applies globally to every test. One line, zero ambiguity.

**What was NOT the cause:**
- Session driver (`array` vs `redis`) — this was correct
- User factory / password hashing — working fine
- Routes / controllers — all correct
- Database setup — working fine

The 419s were masking everything else. Once CSRF was removed from the test layer, all 14 failures cleared in one go.

## Does Media Processing Showcase Queues Better?

**Yes, and here's why it's not even close:**

With CSV import, the queue is an implementation detail — the user submits a file and eventually the data appears. The async nature isn't obvious unless you have a very large file or artificially slow it down.

With media processing, the queue *is* the user experience:
- The user uploads an image and can visibly see it's being processed
- Multiple jobs (resize to 3 sizes, generate thumbnail, watermark) are organically parallel or chained
- A failed job (corrupt file, wrong format) is a real, demonstrable scenario
- Progress feedback is a natural expectation — so Broadcasting fits without feeling forced

**For an assignment where queues are the core requirement**, media processing lets you demonstrate:

| Queue Feature | Media Processing | CSV Import |
|---|---|---|
| Job dispatching | Natural | Yes |
| Job chaining | Natural (resize → thumbnail → notify) | Forced |
| Job batching | Natural (multiple output sizes) | Core strength |
| Failed jobs | Realistic (bad file) | Realistic (bad row) |
| Queue priorities | Logical (thumbnails > originals) | Less obvious |
| Real-time progress | Expected by users | Unusual |
| Observable async | Inherent | Requires large data |

CSV's strength is `Bus::batch()` for chunk processing — that's genuinely clean queue usage, but it's one pattern. Media processing exercises more of the queue API naturally.

## Justification: Iterative Incremental for This Project

### 1. The Architecture Has Natural Increment Boundaries

This project has a strictly layered dependency chain:

```
Infrastructure → Domain Model → Queue Jobs → Broadcasting → UI
```

Each layer is a prerequisite for the next. You cannot test Broadcasting without jobs. You cannot test jobs without the domain model. You cannot test the domain model without infrastructure.

Iterative Incremental maps directly onto this — each increment is a layer that is **built, tested, and verified before the next begins**. The architecture itself prescribes the methodology, not the other way around.

No other methodology exploits this structure. Kanban ignores it. Scrum cuts across it with arbitrary sprint boundaries.

---

### 2. It Controls the Primary Risk of AI-Assisted Development

The failure mode unique to AI pair programming is this: the agent generates code quickly across multiple layers simultaneously, integration is attempted late, and failures are hard to isolate because too many moving parts were introduced at once.

Iterative Incremental enforces a discipline that directly counters this:

> **Nothing moves forward until the current increment is verified.**

Each increment produces a stable, testable checkpoint. If Broadcasting is broken, you know the queue jobs beneath it were sound — the problem is isolated to the current increment, not buried across the entire codebase.

---

### 3. It Is Honest About What We Know Now vs Later

Waterfall requires fully specifying the UI before writing infrastructure. That's a false assumption — implementation of the queue layer will reveal constraints that affect the Livewire component design. We don't know those constraints yet.

Iterative Incremental acknowledges this honestly. Later increments are planned loosely and refined as earlier increments teach us more. It is the only lifecycle that treats **learning as a first-class project output**.

---

### 4. It Pairs Naturally With TDD

Test Driven Development requires a clear, small target to write a test against before implementation. Increments provide exactly that — a bounded scope with defined entry and exit criteria.

In Kanban or Scrum, TDD can coexist but there is no structural pressure to maintain it. In Iterative Incremental, the exit criterion for each increment **is** the passing test suite. The two methodologies reinforce each other.

---

### 5. It Protects the Assignment Deliverable

If time runs short — which is always a risk in a technical assignment — Iterative Incremental guarantees you have a **working, demonstrable system** at every checkpoint, not a partially connected system that runs nothing end to end.

After Phase 2 you can demo image upload and persistence. After Phase 3 you can demo queue processing and Horizon. After Phase 4 you have the full real-time system. Each phase is independently demonstrable.

Kanban without lifecycle structure cannot make this guarantee. Neither can Scrum if a sprint cuts across multiple layers.

---

### 6. It Is Appropriate for the Team Size

Scrum's ceremonies — standups, sprint reviews, retrospectives, backlog refinement — exist to solve communication and coordination overhead in teams of 5–12 people. This team is two participants, one of whom has perfect recall and zero communication latency.

The overhead is pure waste. Iterative Incremental gives you the rigour of a defined lifecycle without the ceremony tax.
